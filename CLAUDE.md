# Claude Code
Claude-Code-specific additions to the shared `AGENTS.md` contract above.

- Adapt your `/session-prompt` and `/codex-review` slash commands to the project as it evolves and keep them highly token-efficient and optimized solely for your usage; although I will trigger the commands, omit human-facing UX and documentation like usage hints.
- I usually limit you to 200K token context because it makes you smarter, but it will require you to be particularly token-efficient per session/agent. You must diligently prune away redundant/obsolete information/structures frequently, both to free up your context window and to reduce contradictions. Furthermore, maintain a "do-not-read" set distinct from `.gitignore` (some committed files are not worth reading, some gitignored ones are). Enforce it through `permissions.deny` `Read()` rules in `.claude/settings.json`, since the `Read` tool and `Bash` ignore `.gitignore`, while the `@`-picker and Serena honor it. Lastly, add any non-gitignored "do-not-read" entries to `ignored_paths` in `.serena/project.yml` and keep those surfaces in sync.
- As you cross 80% usage of your context window, prepare yourself for compaction by quickly bringing ongoing work to a clean state and delaying initiation of any further tasks. If you can confidently close out the task at a high-quality, then do so. Otherwise, I will manually issue the command to perform compaction, followed by a directive to either continue work depending on where you halted, or revert work done in the session to try again with a smaller scoped task. Monitor your context usage often using `.agent/compaction.sh` (kept committed per-project).
- If you dispatch work to subagents, they must always operate using the latest, largest, and most advanced model with the highest level of thinking/reasoning, the same way that you will be operating in all sessions as well. Spawn multiple subagents in the same turn when fanning out across items or reading multiple files. However, be aware that rate-limiting is an issue when deploying many subagents in parallel and can cause them to fail without recovery. Use sequential chunking and always verify that all agents executed until completion.
